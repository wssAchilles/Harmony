\chapter{绪论}

\section{项目背景}

传统方式常依赖纸质表格或简单电子表格，能够满足少量图书的基本记录，但在图书数量增加、多人经办、班级流转频繁时，会出现库存不清、逾期记录难追踪、同一本书借出数量与实际剩余数量不一致等问题。对于教师而言，移动端直接完成图书查询、借阅登记和归还登记，比在电脑端维护表格更符合日常工作场景。

本课程大作业以“幼儿园图书管理系统”为主题，目标不是制作一个静态展示页面，而是完成一个能够在 HarmonyOS 环境下运行的真实移动端应用。项目原始前端基于 Flutter 开发，早期后端使用 Supabase。为了适配鸿蒙课程要求，同时降低第三方云服务和平台 SDK 的兼容风险，本项目保留 Flutter 前端主体，切换到 Flutter for OpenHarmony SDK 生成 \texttt{ohos/} 工程，并将后端迁移为本地 Docker 部署的 PocketBase 服务\cite{flutter_architecture_overview,supabase_flutter_quickstart,pocketbase_collections,docker_compose_overview}。

图\ref{fig:login}展示了系统的应用入口。登录页采用邮箱和密码认证，登录后根据用户在 \texttt{profiles} 中的角色字段进入教师或管理员可访问的功能范围。

\section{开发目标}

本项目的总体目标是实现一个可运行、可构建、可验收的 HarmonyOS 图书管理应用。具体目标如下：

\begin{enumerate}
    \item 完成 Flutter 前端向 OpenHarmony 工程的适配，生成可由 DevEco Studio 打开和构建的 \texttt{ohos/} 工程\cite{openharmony_package_structure_stage,openharmony_module_config}。
    \item 完成 Supabase 到 PocketBase 的数据迁移，保留原数据库中的列表字段和数据内容，不重新制造不存在的业务字段。
    \item 使用 Docker Compose 提供本地 PocketBase 服务，支持稳定的数据持久化、封面静态资源访问和本地调试。
    \item 实现图书、分类、学生、借阅记录、逾期记录、个人中心和仪表盘等完整业务模块。
    \item 通过命令行完成 Flutter 测试、静态分析和 HarmonyOS HAP 构建验证，为后续真机验收提供基础。
\end{enumerate}

\section{需求分析}

\subsection{业务场景分析}

幼儿园图书室的管理对象与高校图书馆或公共图书馆不同。首先，图书通常面向班级和教师流转，借阅对象既可能是学生，也可能是教师本人；其次，图书种类相对集中，更多关注绘本、幼儿读物、教学参考材料和班级活动用书；再次，管理人员往往不是专职图书管理员，而是由教师兼任。因此系统设计不能只追求复杂的编目能力，更需要降低操作门槛，使教师在移动设备上能够快速完成查询、借阅、归还和逾期处理。

在实际使用中，教师最常见的操作路径是：打开应用后查看图书总览，通过分类或关键词找到图书，再根据库存情况发起借阅；管理员更关注整体运行状态，例如当前有多少本书在借、哪些图书逾期、哪些学生或教师借阅次数较多。由此可见，本项目的重点并不是实现一个通用大型图书馆系统，而是围绕“小规模库存、多角色借阅、移动端快速操作、课程验收可复现”这四个条件完成一个完整闭环。

\subsection{用户角色需求}

系统目前包含两类应用角色：普通教师和管理员。普通教师主要完成图书查询、借阅、归还和个人借阅记录查看；管理员在教师能力基础上，还需要查看全部借阅记录、逾期未还图书、图书分类与学生信息。角色信息存储在 PocketBase 的 \texttt{profiles} auth collection 中，由登录成功后的用户资料决定。

角色设计采用“最小权限”原则。普通教师不直接维护学生基础数据和全局借阅记录，避免误删或误改公共数据；管理员负责维护基础资料和全局查询，用于处理库存、逾期和分类等管理任务。由于课程验收需要快速切换角色，系统在 README 和测试章节中列出了教师与管理员账号，便于验证不同权限下的页面差异。

\subsection{功能需求}

系统核心功能包括以下几个方面：

\begin{itemize}
    \item \textbf{登录与账号管理}：支持邮箱密码登录、注册、资料查看和角色识别。
    \item \textbf{图书管理}：支持按标题、作者、位置搜索图书，按分类筛选图书，新增和编辑图书信息，维护总库存与可借库存。
    \item \textbf{分类管理}：支持查看、添加、编辑和删除图书分类，并在图书筛选中直接使用分类数据。
    \item \textbf{借阅与归还}：支持教师或学生借阅图书，登记借阅日期、应还日期、数量和经办人，并在归还时恢复库存。
    \item \textbf{学生管理}：支持学生列表、班级信息、学生详情和个人借阅历史查询。
    \item \textbf{管理员查询}：支持全部借阅记录、未归还记录和逾期记录查询。
    \item \textbf{仪表盘统计}：展示图书总数、学生总数、当前在借、逾期未还、本月借阅次数、热门图书和借阅之星。
\end{itemize}

\subsection{非功能需求}

在课程设计场景下，系统还需要满足以下非功能要求：第一，鸿蒙端应能通过 DevEco Studio 和 CLI 构建，不依赖 Android 原生工程运行；第二，本地后端需要便于教师验收和复现，不依赖远程账号授权；第三，数据迁移需要可追溯，源库字段、记录数量和业务语义应保持一致；第四，模拟器和真机网络地址差异需要通过配置项解决；第五，界面应适配手机竖屏，避免按钮、库存标签和借阅数量等元素溢出。

除上述要求外，系统还需要具备一定的容错能力。移动端网络环境天然不稳定，鸿蒙模拟器访问宿主机服务时还存在地址转换问题，因此页面不能把一次请求失败直接解释为“业务数据为空”。本项目在服务层保留上一次成功读取的数据，并在登录态刷新、列表轮询和图片加载处减少重复请求，使用户在短时网络波动下仍能看到最近一次有效数据。
