\chapter{总结与展望}

\section{工作总结}

本文完成了一套面向幼儿园图书室场景的鸿蒙移动端图书管理系统。系统保留 Flutter 前端主体，通过 Flutter for OpenHarmony 生成 \texttt{ohos/} 工程，并在 ArkTS EntryAbility 中嵌入 Flutter 页面，实现了从原 Flutter 应用到 HarmonyOS 工程的平滑迁移。应用端已经覆盖登录、图书管理、分类管理、学生管理、借阅归还、逾期查询、管理员借阅记录和仪表盘统计等核心功能。

后端方面，系统将 Supabase 迁移为 PocketBase + Docker 的本地部署方案。迁移严格遵循原数据库字段和数据内容，围绕 PocketBase 的系统 id、auth collection 和 nullable 字段限制做最小适配；同时通过 Docker named volume 保存 \texttt{/pb\_data}，通过 \texttt{pb\_public/covers} 提供封面图片静态资源。相比依赖远程 Supabase 服务，本地 PocketBase 后端在课程验收中更易启动、更易复现，也更便于定位网络与数据问题\cite{supabase_auth_docs,pocketbase_collections,docker_compose_volumes}。

在工程稳定性方面，系统修复了模拟器访问宿主机地址、重复鉴权刷新、偶发空列表、图片路径解析和小屏幕界面溢出等问题，并通过 Flutter 分析、单元测试和 HarmonyOS HAP 构建验证，形成了完整的开发、迁移、部署和构建闭环。

从课程目标看，本项目覆盖了移动应用开发中的多个关键环节：界面设计、状态管理、网络请求、数据持久化、权限配置、跨平台构建和部署验证。尤其是在鸿蒙适配部分，项目没有停留在 Android 或 Web 运行效果，而是生成了真实 \texttt{ohos/} 工程，并围绕 HarmonyOS SDK、DevEco Studio、HAP 构建和模拟器网络进行了实际调试。

从软件工程角度看，项目还体现了需求拆分、模块化设计、数据迁移、测试验证和问题复盘。数据库从 Supabase 切换到 PocketBase 并不是简单替换 URL，而是涉及字段映射、认证模型、图片资源、Docker volume、移动端配置和服务层容错的一组系统性修改。最终系统能够在本地后端和鸿蒙端之间形成稳定闭环，说明改造方案具有可实施性。

\section{不足之处}

受课程项目时间限制，系统仍存在一些可以继续优化的地方：

\begin{itemize}
    \item 当前 OpenHarmony 下 \texttt{shared\_preferences} 插件可能存在适配差异，因此客户端提供了内存 auth store 回退方案；后续可接入更稳定的鸿蒙本地持久化插件。
    \item 目前 PocketBase 运行在本机 Docker 中，适合课程验收和局域网测试；若用于真实园所，需要进一步设计备份、权限、HTTPS 和运维策略。
    \item 封面图片当前由静态目录提供，已经解决显示问题，但尚未实现完善的移动端上传、压缩和图片管理流程。
    \item 借阅记录已有数量字段和库存联动，但仍可进一步加强并发借阅场景下的事务一致性。
\end{itemize}

此外，当前测试主要依赖开发者手动操作和命令行构建验证，自动化端到端测试仍不充分。由于鸿蒙模拟器的自动化测试环境搭建成本较高，本文没有实现完整的 UI 自动化脚本。后续如果项目继续演进，可以将服务层逻辑、数据 mapper、库存计算和日期判断纳入更系统的单元测试，降低后续修改引入回归的风险。

\section{未来展望}

后续可以从以下方向继续扩展系统：

\begin{enumerate}
    \item 接入条形码或 ISBN 扫描，减少教师手工录入图书信息的成本。
    \item 增加借阅到期提醒、班级借阅报表和月度统计导出功能。
    \item 为管理员提供 PocketBase 数据备份、恢复和健康检查页面。
    \item 针对 HarmonyOS 真机完善本地存储、图片选择、权限申请和网络错误提示。
    \item 将教师、管理员和园所负责人权限进一步细分，支持更细粒度的数据访问控制。
\end{enumerate}

在部署方面，未来可以把 PocketBase 从本机 Docker 迁移到园所内网服务器或轻量云服务器，并为移动端配置 HTTPS 域名。这样教师真机不再依赖开发者 Mac 的局域网地址，数据也可以由统一服务器集中保存。在权限方面，可以进一步利用 PocketBase collection rules，把教师、管理员和园所负责人区分为不同访问级别，使前端展示权限和后端数据权限保持一致。

在鸿蒙能力方面，后续可以继续接入系统相册、扫码、消息通知和本地安全存储等能力。例如通过扫码快速录入图书编号，通过本地安全存储保存登录 token，通过通知提醒即将逾期的图书。这些能力会进一步体现 HarmonyOS 原生生态价值，但也需要逐项验证对应插件或 ArkTS 原生能力的可用性。

总体来看，本项目不仅完成了一个可运行的幼儿园图书管理应用，也体现了鸿蒙移动端适配、数据库迁移、本地化部署、业务数据一致性和软件工程文档化等课程要求。
