\section{界面适配测试}

界面测试重点关注手机竖屏下的元素溢出、点击区域和滚动体验。由于鸿蒙模拟器的状态栏、导航栏和侧边控制条会占用视觉空间，页面实际可用宽度比桌面预览更紧张。图书卡片、个人中心借阅卡片和管理员记录卡片都进行了窄屏检查。

主要调整包括：将并列文本改为可换行布局；将还书按钮和数量标签设置为固定尺寸；将长标题、日期和班级信息放入弹性区域；将分类筛选放入横向滚动容器；对底部导航预留安全区域。调整后，页面在模拟器中不再出现 Flutter 的红色 overflow 提示，关键按钮也不会被数量标签或日期文本遮挡。

\section{后端运行验证}

PocketBase 服务验证包含三个层次。第一，Docker 层确认容器正在运行并暴露 \texttt{8090} 端口；第二，PocketBase 层确认管理后台可以打开、collection 存在且记录数量正确；第三，移动端层确认模拟器可以通过 \texttt{10.0.2.2:8090} 登录并读取图书。三层都通过后，才能说明后端不仅“启动了”，而且“能被鸿蒙端使用”。

常用检查命令如下：

\begin{lstlisting}[style=codexcode]
docker compose ps
curl http://127.0.0.1:8090/api/health
curl http://127.0.0.1:8090/covers/book_001.png
\end{lstlisting}

{\RaggedRight\emergencystretch=2em 其中图片检查用于验证静态资源目录是否正确挂载。若 API\hspace{0pt} 正常但图片不显示，通常应优先检查 \texttt{pb\_public/covers}\hspace{0pt} 是否存在文件、数据库中的 \texttt{cover\_image\_url}\hspace{0pt} 是否为相对路径，以及前端是否把相对路径拼接到了正确的 base URL。\par}

\begin{lstlisting}[style=codexcode]

\section{稳定性问题分析}

项目调试过程中出现过两类典型问题。第一类是模拟器无法连接后端，表现为 \texttt{Connection refused} 或状态码为 \texttt{0}。原因是模拟器内的 \texttt{127.0.0.1} 指向设备自身，而 PocketBase 实际运行在 Mac 宿主机。修复方式是默认使用 \texttt{10.0.2.2:8090}，真机再切换为 Mac 局域网 IP。

第二类是偶发空列表或统计数据异常。排查后发现不是业务数据丢失，而是重复鉴权刷新、过密轮询、Docker bind mount 上的 SQLite I/O 抖动和瞬时请求失败叠加导致。SQLite WAL 会在数据库文件旁维护额外日志文件并涉及检查点写回，因此将数据库文件放在 Docker 管理的 volume 中更符合本项目的稳定性目标\cite{sqlite_wal,docker_compose_volumes}。修复后系统采取了以下策略：

\begin{itemize}
    \item \texttt{AuthGate} 不再在每次 build 后重复触发 \texttt{authRefresh}。
    \item 列表轮询间隔收敛到 30 秒，减少高频请求。
    \item \texttt{BookService} 和 \texttt{DashboardService} 保留上一次成功数据，瞬时失败时不立即显示空态。
    \item PocketBase 的 \texttt{/pb\_data} 使用 Docker named volume，不再直接放在 macOS bind mount 上。
\end{itemize}

这些优化使系统更适合鸿蒙模拟器和真机环境下的连续操作，也让课程验收时的后端部署更容易复现。

\section{问题定位复盘}

项目调试过程中的问题具有明显的跨平台迁移特征。白屏问题通常不是页面没有写，而是入口 Ability、Flutter 容器或资源加载链路未完全启动；登录连接失败不是账号错误，而是模拟器访问宿主机地址错误；图片不显示不是图片组件失效，而是静态路径和 base URL 不匹配；偶发空列表也不是数据库数据丢失，而是请求失败后页面错误地使用空数组覆盖了原有数据。

通过这些问题可以总结出一条有效的定位顺序：先看后端是否启动，再看地址是否能从当前设备访问，再看接口返回字段是否符合模型，再看页面是否正确处理加载态和错误态。该顺序比直接修改 UI 更可靠，也符合移动端前后端分离系统的调试规律。

对于数据库稳定性问题，最终方案没有简单地增加重试次数，而是同时减少不必要请求、调整 Docker volume、修正服务层缓存和明确查询字段。这样处理后，系统在模拟器连续切换页面时更稳定，也能避免“偶发查不到数据”影响人工验收。
